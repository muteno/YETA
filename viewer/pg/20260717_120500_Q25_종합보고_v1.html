항목내용

파일명 감정 표기안 써도 됨(확정). 시스템은 파일명을 전혀 안 읽고 manifest의 버킷(어느 폴더에 있나)만 봄. 윈터 폴더의 glaring_03.png류 영문 감정명은 현 시스템 미사용 — 앞으로 수집분은 아무 이름(숫자)으로 폴더에 붓기만 하면 됨.

③ 맥락·감정 따라 배경 변화(몰입) — 소견 답변+반영

방향 자체에 전적으로 동의 — 그리고 그 뼈대가 이미 앱에 있음. 현재 우선순위: 난입 연출 > 오프닝 시그니처(첫/오랜만 처음 몇 마디) > 장소 씬(시간대·계절·날씨 종속) > 무드 씬 > 기본. 즉 "처음엔 시그니처 → 평소엔 장소 → 감정이 튀면 무드"가 정확히 구현된 상태고, 이번에 모드축(⑧₁)이 보강되면서 시즌 캐릭터는 [모드 → 무드 → 로테이션] 3단이 됨.

다음 단계로 값진 순서(제안): ⑴ 안드로 사진도 감정 폴더 분화(warm_android/ 등 — 생성기에 한 줄) ⑵ 페르소나 확립 후 시그니처 배경을 "관계 레벨"로 교체(LV 게이트 재활용) ⑶ 대화 속 장소 언급을 러너가 <<PLACE>>류 태그로 흘려 장소 씬과 직결. 전부 기존 축 위에 얹는 것이라 새 기틀이 필요 없음.

④ 상단 네비 소환↔원거리 간격 통일 해결

스샷 픽셀 실측→원인 = 스테일 마진 6px 상시 적용→마진 2종 제거→헤드리스 렌더 실측 검증
항목내용
원인(실측)원거리 배지(평소 숨김)용 규칙 .yh-remote + .yh-r { margin-left:6px }가 배지가 숨어 있어도 항상 적용(CSS 형제 선택자는 display:none을 안 가림) → 소환↔원거리 14px vs 원거리↔전화 6px. 죽은 margin-left:auto 2곳도 동반 정리.
조치(전→후)전: 박스 간격 14 vs 6 (글리프 광학 ≈29.7 vs 24.2px) → 후: 스테일 마진 제거 = 글리프 광학 간격 23.7 vs 24.2px(0.5px 이내) — 헤드리스 크로미엄 픽스처로 bounding box 실측 검증. 원거리 픽토가 전화 쪽 간격과 같은 리듬으로 정렬됨.
리스크원거리 배지(문자 '원거리' 필)가 뜨는 화면에서도 간격 = 헤더 공용 gap 8px 단일 — 배지 유무에 따라 전화 위치가 6px 튀던 부수 현상도 함께 소멸.

⑤ '#' 상황 서술로 상황 흐름 제어 가능한가 가능 — 계약 명문화

항목내용
현재 구조이미 가능 — # 상황 턴은 러너에서 "— 상황(유저 연출): … —"로 대사와 격리 주입되고, 프롬프트가 "무대 지시 = 장면의 사실로 수용"을 요구함. (데스크가 세웅쪽으로 이동한다) 같은 3자 행동 연출도 이 축.
보강(이번)출력 계약에 인라인 연출 조항 추가: 메시지 안 (괄호 한 줄)·*별표* 줄 = "방금 무대에서 실제로 일어난 일(인물 이동·행동·상황 변화)"로 수용 + 다른 주민의 행동도 사실로 수용 명시. 단 캐릭터·말투·규칙을 바꾸라는 연출만 무시(세계관 보호 — 기존 보안 결 유지).
한계(정직)연출이 캐릭터 카드의 금기(예: 루시에게 억지로 음식 먹이기)와 충돌하면 인물이 세계 안에서 저항하는 쪽으로 반응함 — 이건 버그가 아니라 카드 계약. 완전 신 권한(God-mode)을 원하면 별도 신호 축 설계 필요(후속 제안 가능).

⑥ #입력 = 별도 전송 말고 (괄호)로 한 메시지 합성 해결

현행 = 구간별 분리 발송→혼합 시 한 메시지로 합성→# Enter = 전송 아님·구간 닫기→(괄호) 줄 = 지문 결 렌더
항목내용
동작(후)"안녕?" 입력 → # → "루시가 그런 나를 바라보며 인상을 쓴다" → Enter = 전송이 아니라 괄호 구간 닫기 → 이어서 다음 말 입력 → 전송 버튼(또는 대사 상태 Enter) = 한 메시지:
안녕?
(루시가 그런 나를 바라보며 인상을 쓴다)
그 다음에 내가 적는 메세지
괄호 줄은 버블 안에서 이탤릭·톤다운(.yn 지문 결 = 기존 *지문*과 동일 문법) 렌더. #을 다시 눌러도 동일(토글 유지).
경계 유지상황만 보낸 턴 = 종전 그대로 회색 지문 라인(sc 턴 · 버블 없음) — 화면 중앙 연출 문법 보존. 러너 전달도 이원화: 단독 = "— 상황(유저 연출) —" 블록 · 혼합 = 메시지 속 (괄호) 줄(⑤ 계약이 수용).
전후 비교전: #구간이 항상 별도 턴으로 찢어져 나감 + #상태 Enter = 즉시 전송(오발 잦음) → 후: 의도대로 묶어서 한 방에, 전송 시점은 온전히 사용자 손에.
리스크·완화기존 습관(# 쓰고 Enter로 바로 전송)이 한 단계 늘어남(Enter 한 번 더) — 대신 오발송 소멸. 모바일 키보드 '보내기' 키도 동일 규칙.

⑨⑩ 쓰던 답장 새로고침 → 60% 기준 낑김/재생성 해결

원인 실측(둘 다 있었음)→60% 판정 = 생성 경과시간 비율→≥60% 유지+mut 낑김→<60% 폐기·새 맥락 재생성→화면: 보낸 뒤에도 draft 유지(mut)
항목내용
원인(실측 2겹)㉮ 화면: 내가 보내는 순간 '읽는 시간' 게이트가 리셋돼 자라던 부분 답장이 즉시 사라짐(= "새로고침"으로 보이던 것) ㉯ 데이터: 생성 완료본은 원래 내 새 메시지 앞에 끼워지는데, 화면이 숨겼다가 나중에 통째로 나타나니 "말을 다시 시작했다"처럼 보였음.
60% 판정(⑩ 확정)운영자 직관 그대로 시간 비율로 구현: (내 메시지 도착시각 − 생성 시작) ÷ (생성 완료 − 시작) ≥ 60% → 유지: 답장에 ov 표식 → 대화에 mut 컬러로 낑김(행위·서술이 원래 mut인 것과 동일 톤 = 운영자 지정). < 60% → 무기록 폐기 → 러너 웜 루프가 내 새 메시지까지 묶어 처음부터 다시 답함(맥락 정합).
화면(⑨)보내는 순간 draft를 숨기지 않고 mut 톤으로 계속 자라게 유지 — "하던 말은 이어서". 완료되면 같은 자리(내 메시지 위)에 확정 스왑.
전후 비교전: 겹치면 draft 증발 → 한참 뒤 두 답이 순서 애매하게 등장 → 후: ≥60%는 눈앞에서 끝까지 말하고 mut로 남음 + 새 메시지 답이 이어짐 / <60%는 깔끔히 접고 통합 답 1개.
